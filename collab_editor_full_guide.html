What you're building
CollabCode — Google Docs for code
Multiple users edit the same file simultaneously from different browsers. Edits appear in real-time for all users, cursors are tracked per user, conflicts are resolved automatically without any locking, and the document always converges to the same state. Syntax highlighting, language selection, and session history included.
The core problem — why this is hard
Problem Concurrent edits break naively
User A and User B both start with "Hello". A inserts "World" at position 5. B deletes "Hello" at position 0. If you apply both edits sequentially using raw character positions, one edit corrupts the other. This is called the lost update problem and it's why you can't build this with a simple "send the diff via WebSocket" approach.
Solution CRDTs — conflict-free replicated data types
A CRDT is a data structure designed so that concurrent operations always merge correctly without coordination. You use the Yjs library which implements a CRDT called Y.Doc. Instead of tracking character positions (which break under concurrency), Yjs tracks every character as a unique node with a logical clock ID. Merging always works, always produces the same result, regardless of order operations arrive.
HLD — 4 layers
Layer 1 Client — React + Monaco Editor + Yjs
Monaco is the same editor VS Code uses. Yjs binds to it via y-monaco so that every keystroke becomes a CRDT operation, not a raw string diff. Users see each other's cursors as coloured carets. The client connects to the server via WebSocket (y-websocket provider) and optionally persists to IndexedDB (y-indexeddb) so the doc loads offline too.
Layer 2 Sync server — Node.js + y-websocket
A thin WebSocket server that receives CRDT update messages from any client and broadcasts them to all other clients in the same room. It doesn't understand the document contents — it just relays binary Yjs update blobs. Optionally persists the latest doc state to Redis so new joiners don't have to replay all history.
Layer 3 Persistence — PostgreSQL + Redis
PostgreSQL stores room metadata, user accounts, and session history. Redis stores the latest serialised Y.Doc state per room (binary blob, updated on every change) so new users joining an existing session get the current state instantly without replaying all operations from the beginning.
Layer 4 Awareness layer — cursor + presence
Yjs has a built-in "awareness" protocol on top of WebSocket. Each client broadcasts its cursor position, username, and colour. All other clients receive this and Monaco renders coloured cursor decorations in real-time. This is separate from the document sync — awareness data is ephemeral and lost on disconnect.
The concept — without the maths
Why character positions fail
Imagine two users. Both see: Helo. User A inserts 'l' at index 2 → Hello. User B inserts '!' at index 4 → Helo!. If A's update arrives first and shifts characters right, B's index 4 now points to 'o' instead of end. B's '!' lands in the wrong place. Every simple diff-based approach has this flaw.
How Yjs fixes it — unique IDs, not positions
Instead of "insert at position 5", Yjs says "insert after the node with ID (clientA, clock:7)". IDs are stable — they don't shift when other characters are inserted. A character always knows what it comes after, regardless of what happened concurrently. When two clients insert at the same position simultaneously, a deterministic tiebreaker (client ID) decides order. Same inputs always produce the same result on every client.
CRDT vs OT — the question interviewers love
Operational transformation (OT) — the older approach
Google Docs used OT. The idea: transform an operation based on what happened concurrently. "Insert at 5" becomes "insert at 6" if someone inserted before position 5 first. Works, but requires a central server to order all operations. Hard to implement correctly, especially for rich text. Google has a team dedicated to it.
CRDTs — the modern approach (what you're using)
No central ordering needed. Each client applies operations independently and the state always converges. Yjs's algorithm (called YATA) is peer-to-peer friendly — works even offline. Slightly higher memory use than OT, but this is negligible for code files. Figma, Linear, and most modern collaborative tools use CRDTs.
Yjs key concepts you must know
1

Y.Doc

The root shared document. Every client has one. They sync by exchanging binary update messages.

2

Y.Text

A shared text type inside a Y.Doc. Monaco's y-monaco binding listens to this and applies changes. All concurrent edits go through Y.Text — never touch Monaco's model directly.

3

Provider

How the Y.Doc syncs with others. y-websocket provider connects to your server and sends/receives update blobs. y-indexeddb provider persists to the browser so the doc survives page refresh offline.

4

Awareness

Yjs's ephemeral state layer. Use it for cursor positions, usernames, online status. Awareness data is NOT part of the document and NOT persisted — it's just broadcast to current peers.

Workflow 1 — user joins a room and gets current doc state
1

User opens /room/abc-123

React renders Room.jsx. useYjs hook fires — creates a new Y.Doc and connects a WebsocketProvider to ws://sync-server/abc-123.

2

Sync server checks Redis for existing state

If a Y.Doc state blob exists for room abc-123 in Redis, the server sends it immediately as a "sync step 1" message. The client applies it and the editor fills with the current document content instantly.

3

MonacoBinding attaches

Once the Y.Doc has initial state, MonacoBinding links Y.Text to Monaco's model. From this point, every keystroke in Monaco creates a Y.Text operation, and every remote operation updates Monaco.

4

Awareness broadcast

Client sets its cursor state (username, colour, cursor position). All other connected clients receive it and render a coloured cursor decoration in Monaco at the correct position.

Workflow 2 — concurrent edit (the core CRDT flow)
1

User A types 'x' at position 3

y-monaco intercepts the Monaco change event. Creates a Y.Text insert operation with a unique ID (clientA, clock:15). Applies it locally to the Y.Doc immediately — no waiting. Monaco shows 'x' instantly.

2

Y.Doc serialises the update

Yjs encodes the operation as a compact binary blob (typically 20–50 bytes for a single character). The WebsocketProvider sends this blob to the sync server.

3

Sync server broadcasts to all room members

The sync server receives the binary blob and forwards it to every other WebSocket connection in room abc-123. It does not parse the blob — just relays bytes.

4

User B's Y.Doc applies the remote update

Yjs merges the incoming operation using CRDT rules. If User B also typed at the same position simultaneously, Yjs resolves the conflict deterministically. Both documents end up identical. Monaco re-renders.

5

Redis updated

Sync server serialises the full Y.Doc state and writes it to Redis under key room:abc-123. New joiners get this state immediately.

Workflow 3 — session snapshot (history feature)
1

User clicks "Save snapshot"

Client calls POST /api/rooms/:id/snapshots with the current plain text content (ytext.toString()).

2

Server saves to PostgreSQL

The snapshot row is inserted with the full code text, timestamp, and user ID. Snapshots are read-only historical records — you can view them but not restore destructively.

3

History panel shows all snapshots

GET /api/rooms/:id/snapshots returns all saved snapshots. User can click any to view the code at that point in a read-only Monaco instance side-by-side with the live editor.

Questions you will definitely be asked

"Walk me through your project."

Say: "I built a real-time collaborative code editor — think Google Docs but for code, using Monaco which is the VS Code editor. The interesting problem was handling concurrent edits. If two people type at the same position simultaneously, naive string diffing corrupts the document. I solved this using Yjs, which implements a CRDT called YATA — instead of tracking character positions, every character gets a unique ID and knows what it comes after. Concurrent edits always merge correctly, no locking needed, and the document converges to the same state on every client."

"What is a CRDT and how does it work?"

Say: "CRDT stands for conflict-free replicated data type. It's a data structure designed so that concurrent operations always merge correctly without coordination. In Yjs specifically, each character isn't stored at a position — it's stored as a node with a unique ID (clientId + logical clock) that points to what it comes after. When two clients insert at the same spot simultaneously, the CRDT uses a deterministic tiebreaker — the client ID — to decide order. Any client that applies both operations in any order ends up with the same result. It's mathematically guaranteed to converge."

"CRDT vs Operational Transformation — what's the difference?"

Say: "OT is the older approach — Google Docs uses it. You transform each operation based on what happened concurrently. 'Insert at 5' becomes 'Insert at 6' if someone inserted before position 5 first. It works but requires a central server to order all operations. CRDTs don't need central ordering — each client applies operations independently and the state always converges. Yjs is peer-to-peer friendly, works offline, and is what most modern tools like Figma and Linear use."

"How do you handle cursor positions for other users?"

Say: "Yjs has a built-in awareness protocol that's separate from document sync. It's a lightweight pub-sub layer for ephemeral state — things that aren't part of the document. Each client broadcasts its cursor position, username, and colour. Other clients receive this and I use Monaco's deltaDecorations API to render a coloured cursor caret at that position. Awareness data is not persisted — it disappears when the user disconnects."

"What happens when a user joins a room that already has content?"

Say: "The sync server stores the full serialised Y.Doc state in Redis on every update. When a new WebSocket connection opens for a room, the server fetches the binary blob from Redis and sends it as the initial sync message. Yjs's applyUpdate merges this state into the client's empty Y.Doc, populating the editor instantly. The client doesn't need to replay all historical operations — just the current state snapshot."

"What would you change to scale this to 10,000 concurrent users?"

Say: "Three things. First, the sync server would need to run as multiple instances behind a load balancer with sticky sessions — clients in the same room must hit the same server, or we need a message broker like Redis pub-sub to relay updates across server instances. Second, instead of storing the full Y.Doc in Redis on every keystroke, I'd debounce it to every 5 seconds. Third, for very large documents, I'd switch from the full-state snapshot approach to delta encoding — only storing and sending the operations that happened since the last known client state."
